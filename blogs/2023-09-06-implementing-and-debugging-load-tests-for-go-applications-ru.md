<h4>Guest Author</h4><p>With more and more people using web and mobile applications for everyday activities such as shopping, listening to music, and watching movies, applications nowadays must handle a tremendous workload. To ensure that your application can handle an expected user load, load testing must be executed before releasing the application.</p><p>However, implementing and debugging your load tests can be tricky. With many simulated requests, it could be daunting to find the root cause of your performance issues by scrolling into pages of the log files. Well, executing the load testing for your application does not need to be like that. This article will teach you how to implement and debug a load test for a Go application running inside a Kubernetes cluster using <a href="https://k6.io/">k6</a> and <a href="https://www.getambassador.io/products/telepresence">Telepresence</a>.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*z1T2XUdOkx4dU6eXDovd6A.png" /></figure><h4><strong>Prerequisites</strong></h4><p>To effectively follow the steps discussed in this, you need to have the following:</p><ul><li>A ready-to-use Linux machine, preferably Ubuntu version 20.04</li><li><a href="https://go.dev/">Go</a>, <a href="https://docs.docker.com/engine/install/ubuntu/">Docker Engine</a>, <a href="https://kubernetes.io/docs/tasks/tools/install-kubectl-linux/">kubectl</a> &amp; <a href="https://code.visualstudio.com/">Visual Studio Code</a> installed on your machine</li><li>A <a href="https://hub.docker.com/">DockerHub</a> account to store the application image</li></ul><h4><strong>What is load testing?</strong></h4><p>When it comes to performance testing, it’s easy to confuse load testing with stress testing. Stress testing is checking how your application behaves when the user load is beyond the volume it can handle. The expected result is your application should fall back to the normal state after the stress testing is done.</p><p>On the other hand, load testing ensures your application can handle an expected number of user requests in a specific time range. Load testing allows you to inspect your application’s behavior under a certain simulated workload. As a result, you will discover your app’s resource utilization issues and address them to ensure the user experience is not impacted by the heavy user load.</p><h4><strong>What is k6?</strong></h4><p><a href="https://k6.io/">k6</a> is a performance testing tool tool created by the Grafana Labs team. k6 is built using Go programming language and uses Javascript for scripting your test scenarios. With k6, you can implement the test for different types of backend services, such as <a href="https://blog.getambassador.io/grpc-vs-rest-apis-key-differences-800ecc66d173">REST APIs</a>, GraphQL APIs, <a href="https://blog.getambassador.io/grpc-vs-rest-apis-key-differences-800ecc66d173">gRPC</a> services, and web sockets. You can also set threshold configs for your tests so that your test will fail if the app reaches the breaking point.</p><h4><strong>About the demo app</strong></h4><p>Let’s have hands-on practice on a demo application to understand how to implement and debug your load application test. The demo application is a blog service that allows users to:</p><ul><li>Create a new blog</li><li>Update an existing blog</li><li>Get all the blogs</li><li>Get the blog content by its id</li><li>Delete the blog by its id</li></ul><p>The blog service will be built using the following:</p><ul><li>Go programming language</li><li>PostgreSQL database to store the app data</li><li><a href="https://www.getambassador.io/resources/building-kubernetes-based-platform">Kubernetes platform</a> to manage and store the blog service and the PostgreSQL database instances</li></ul><p>Now that you have an overview of how the demo application works, let’s move on to the next section to set up the Kubernetes cluster to deploy the app.</p><h4><strong>Set up the Kubernetes Cluster</strong></h4><p>To set up the Kubernetes cluster, you can have two following options:</p><ul><li>Set up from scratch on your infrastructure. This way, you can utilize your existing infrastructure and store all the application data on your servers. However, setting up and maintaining the Kubernetes cluster on your own is complicated and requires in-depth knowledge of how Kubernetes works.</li><li>Set up the Kubernetes cluster using cloud providers’ services such as Amazon Web Service (AWS), Elastic Kubernetes Service(EKS), Google Kubernetes Engine (GKE), or Vultr Kubernetes.</li></ul><p>You can create the Kubernetes cluster using either way, but I would suggest using the Kubernetes cloud provider for easier setup. After creating your own Kubernetes cluster, you should have a Kubernetes configuration file to access the Kubernetes cluster from your machine.</p><p>To access the Kubernetes cluster, open a new terminal in your machine and run this command: export KUBECONFIG=/path/to/the/config/file. Then, run the kubectl get node command to check whether you can access the Kubernetes cluster. You should see a similar output as below:</p><pre>NAME STATUS ROLES AGE VERSION<br />telepresence-k6-c1d2a04201a1 Ready &lt;none&gt; 19h v1.26.2</pre><p>Now that you have a Kubernetes cluster that is ready for use, let’s move on to the next section to deploy the blog app to the Kubernetes cluster.</p><h4><strong>Deploy the application to the Kubernetes Cluster</strong></h4><p>To deploy the app to the Kubernetes cluster, you need to:</p><ul><li>Clone the application code to your machine</li><li>Build the Docker image for your app</li><li>Push your Docker image to DockerHub</li><li>Deploy the PostgreSQL database to the Kubernetes cluster</li><li>Deploy the blog app to the Kubernetes cluster</li></ul><h4><strong>1. Clone the application’s code to your machine</strong></h4><p>The blog application code is at <a href="https://github.com/cuongld2/telepresence-k6-load-test">telepresence-k6-load-test GitHub repository</a>. From the current terminal window, run the following commands to clone the application code:</p><pre>mkdir ~/Projects -p<br />cd Projects<br />git clone https://github.com/cuongld2/telepresence-k6-load-test.git</pre><p>You should see the current structure of your application code as:</p><pre>.<br />├── config<br />├── Dockerfile<br />├── go.mod<br />├── go.sum<br />├── handler<br />├── kubernetes<br />├── main.go<br />└── .vscode<br />├── .env</pre><p>For context,</p><ul><li>The .vscode directory stores the launch.json file. You will use the launch.jsonfile to set up the debugging environment for your application while executing the load testing.</li><li>The config directory stores the db.go file for connecting to the PostgreSQL database.</li><li>The handler directory has the handlers.go file for defining the functions to create, update, retrieve, and delete blogs.</li><li>The kubernetes directory stores all the deployment files for deploying the PostgreSQL database and blog app to the Kubernetes cluster.</li><li>The main.go file is the entry file of the application.</li><li>The .env is the environment file that you will use for accessing the blog app service inside the Kubernetes cluster from your local Visual Studio Code.</li><li>The go.mod and go.sum are for defining the dependencies that your application needs.</li><li>The Dockerfile lists all needed dependencies and commands to build the Docker image for your blog app.</li></ul><p>Now that you have the blog application code on your machine, let’s build the Docker image for the blog service.</p><h4><strong>2. Build the Docker image for the blog service</strong></h4><p>To build the Docker image for the blog app, run the following command below and replace the “username” field with your actual DockerHub’s username.</p><pre>docker build -t username/blog .</pre><p>Wait a few minutes for the application image to be built, then move on to the next section to push the app image to DockerHub.</p><h4><strong>3. Push the app’s image to Dockerhub</strong></h4><p>First, you need to log in to your DockerHub account from the terminal by running the following command:</p><pre>docker login</pre><p>Then, run the following command to push the app image to DockerHub. Don’t forget to replace the username field with your username.</p><pre>docker push username/blog</pre><p>Now that you have the application image in the DockerHub server, let’s move on to deploy the PostgreSQL database to the Kubernetes cluster.</p><h4><strong>4. Deploy the Postgresql database to the Kubernetes Cluster</strong></h4><p>From the current terminal, navigate to the kubernetes/postgresql directory by running this command cd kubernetes/postgresql .</p><p>To have your app data persisted, you need to deploy the PostgreSQL database using Kubernetes Persistent Volume. To create the Persistent Volume for your PostgreSQL database, run the following command: kubectl apply -f postgres-volume.yml. Then, run this command kubectl apply -f postgres-pvc.ymlto create the Kubernetes Persistent Volume Claim to request for storage inside the Persistent Volume.</p><p>The next thing you’ll need to do is to create the PostgreSQL init SQL queries. To do this, run this command kubectl apply -f postgres-init-sql.yml on your terminal and then proceed to deploy the PostgreSQl database to Kubernetes by running the kubectl apply -f postgres-deployment.yml command.</p><p>You need to expose the PostgreSQl database to enable the blog application to access it. To do this, run the kubectl apply -f postgres-service.yml command on your terminal.</p><p>Before proceeding to the next step, let’s confirm that the PostgreSQL database pods and services are running as expected and are up and running by running these commands kubectl get pod and kubectl get service . If all goes as planned, then you should see results similar to the one below:</p><pre>NAME READY STATUS RESTARTS AGE<br />postgresql-5b4669bc56-vkdl8 1/1 Running 0 19h</pre><pre>NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) AGE<br />kubernetes ClusterIP 10.96.0.1 &lt;none&gt; 443/TCP 20h<br />postgresql ClusterIP 10.96.105.126 &lt;none&gt; 5432/TCP 19h</pre><p>Having confirmed that the PostgreSQL database is running in the Kubernetes cluster, let’s move on to the next part to deploy the blog service.</p><h4><strong>5. Deploy the blog service to the Kubernetes cluster</strong></h4><p>From the current terminal, navigate to the kubernetes/blog-app directory by running this command cd ../blog-app. Then open the blog-app-deployment.yml file and replace username in the image:username/blog line with your actual DockerHub username.</p><p>After updating your DockerHub username, go back to your terminal and deploy the blog into the Kubernetes cluster by running the kubectl apply -f blog-app-deployment.yml command.</p><p>The blog app is up and running, but right now, we can only interact with it from inside the Kubernetes cluster. So, let’s make it accessible outside of the cluster by exposing it. To do this, run the kubectl apply -f blog-app-service.yml command.</p><p>Then, proceed to run the kubectl get service to check whether the service we just created is running as expected. If everything works as expected, you’ll get results similar to the one below, showing you that the service creation was successful.</p><pre>NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) AGE<br />blogapp-service LoadBalancer 10.100.190.129 45.77.43.83 8081:32765/TCP 20h<br />kubernetes ClusterIP 10.96.0.1 &lt;none&gt; 443/TCP 20h<br />postgresql ClusterIP 10.96.105.126 &lt;none&gt; 5432/TCP 19h</pre><p>Notice the EXTERNAL-IP value of the blogapp-service service. You will use this when implementing the load test for the blog app service.</p><p>Now that you have deployed the blog app and the PostgreSQL database to the Kubernetes cluster successfully let’s move on to the next section to implement the load test using k6.</p><h4><strong>Implementing the Load Test</strong></h4><p>To implement the load test using k6, first, you must install k6 on your machine. To do so, run the following command:</p><pre>sudo gpg -k<br />sudo gpg - no-default-keyring - keyring /usr/share/keyrings/k6-archive-keyring.gpg - keyserver hkp://keyserver.ubuntu.com:80 - recv-keys C5AD17C747E3415A3642D57D77C6C491D6AC1D69<br />echo &quot;deb [signed-by=/usr/share/keyrings/k6-archive-keyring.gpg] https://dl.k6.io/deb stable main&quot; | sudo tee /etc/apt/sources.list.d/k6.list<br />sudo apt-get update<br />sudo apt-get install k6</pre><p>If you’re using MacOS, Windows, or a non-Debian-based Linux, please refer to the <a href="https://k6.io/docs/get-started/installation/">k6 installation</a> page for details on how to install k6 in different environments.</p><p>Then, run this k6 version command to check whether k6 is successfully installed. On doing this, you’ll see a similar result as below:</p><pre>k6 v0.43.1 (2023–02–27T10:53:03+0000/v0.43.1–0-gaf3a0b89, go1.19.6, windows/amd64)</pre><p>To implement the load test for your blog app service, you need to create a k6 script file. Let’s create a new file named telepresence-blog.js by running the command below</p><pre>cd ../..<br />nano telepresence-blog.js</pre><p>Then copy the following content to the telepresence-blog.js file:</p><pre>import http from 'k6/http';<br />import { check, sleep } from 'k6';<br />export const options = {<br />stages: [<br />{ duration: '5s', target: 10 },<br />{ duration: '5s', target: 5 },<br />{ duration: '5s', target: 0 },<br />],<br />};<br />export default function () {<br />const data = JSON.stringify({ body: 'this is a great blog one',});<br />const domain = &quot;http://45.77.43.83:8081&quot;;<br />const headers = { 'Content-Type': 'application/json'<br />};<br />const res = http.post(domain+'/blog',data,{ headers });<br />check(res, { 'status was 200': (r) =&gt; r.status === 200 });<br />sleep(1);<br />}</pre><p>Let’s go through the test scenario of this k6 script file. In the first two lines, you already imported the http, check, and sleep functions from k6.</p><ul><li>The http function is for creating the HTTP request so that you can interact with the blog APIs</li><li>The check function is for checking whether the HTTP status code of the API equals 200</li><li>The sleep function is for pausing the test for a short period. In real-world scenarios, real users usually wait for one or two seconds before doing the next action to the web app. Using the sleepfunction, you can simulate how real users interact with the service under test.</li></ul><p>The export const optionblock code defines how the simulated users are generated. Here, you defined the users to increase to 10 in 5 seconds. Then, decrease the number of users to 5 in the next 5 seconds. Finally, you decrease the users to 0 in the last 5 seconds.</p><p>The export default functionblock code is to define what the simulated user will do. Basically, you tell the simulated user to make an API request to the API for creating a new blog post. Then you tell k6 to check whether the response status code equals 200. Finally, you tell the simulated user to wait for 1 second before doing the next iteration (After each default function block is executed, if the simulated user is still active, the user will try to execute the whole steps in the code block again).</p><p>Before running the test, you must replace “45.77.43.83” with your actual blog app service external-ip value. Save the file, then run the following command:</p><pre>k6 run telepresence-blog.js</pre><p>You should see a similar test result as below:</p><pre>data_sent………………….: 14 kB 881 B/s<br />http_req_blocked……………: avg=13.89ms min=0s med=0s max=392.07ms p(90)=75.83ms p(95)=79.1ms<br />http_req_connecting…………: avg=13.89ms min=0s med=0s max=392.07ms p(90)=75.83ms p(95)=79.1ms<br />http_req_duration…………..: avg=79.65ms min=68.32ms med=79.31ms max=108.36ms p(90)=84.96ms p(95)=86.57ms<br />{ expected_response:true }…: avg=79.65ms min=68.32ms med=79.31ms max=108.36ms p(90)=84.96ms p(95)=86.57ms<br />http_req_failed…………….: 0.00% ✓ 0 ✗ 78<br />http_req_receiving………….: avg=126.75µs min=0s med=0s max=1.61ms p(90)=467.49µs p(95)=947.57µs<br />http_req_sending……………: avg=14.95µs min=0s med=0s max=187.7µs p(90)=29.66µs p(95)=143.6µs<br />http_req_tls_handshaking…….: avg=0s min=0s med=0s max=0s p(90)=0s p(95)=0s<br />http_req_waiting……………: avg=79.51ms min=68.32ms med=79.08ms max=107.32ms p(90)=84.75ms p(95)=86.57ms<br />http_reqs………………….: 78 5.063333/s<br />iteration_duration………….: avg=1.09s min=1.07s med=1.08s max=1.46s p(90)=1.15s p(95)=1.16s<br />iterations…………………: 78 5.063333/s<br />vus……………………….: 1 min=1 max=10<br />vus_max……………………: 10 min=10 max=10</pre><p>With the above scenario, the average response time of each request is 79.65 milliseconds. There were 78 HTTP requests made, and none of them failed. But there’s a problem with our test script; we forgot to delete the blog post after creating it. Running a number of load test scenarios without cleaning up data will create a lot of unused data and might lead to performance problems for our database. To delete the blog post after it was created, add the following code below sleep(1) line:</p><pre>const deleteRes = http.del(domain+'/blog/'+res.json().id)<br />check(deleteRes, { 'status was 200': (r) =&gt; r.status === 200 });<br />sleep(1);</pre><p>Let’s rerun the test for telepresence-blog.js file by running this command: k6 run telepresence-blog.js. This will return results similar to the one below:</p><pre>✓ status was 200<br />checks…………………….: 100.00% ✓ 80 ✗ 0<br />data_received………………: 12 kB 764 B/s<br />data_sent………………….: 11 kB 663 B/s<br />http_req_blocked……………: avg=16.98ms min=0s med=0s max=415.69ms p(90)=92.29ms p(95)=108.22ms<br />http_req_connecting…………: avg=16.97ms min=0s med=0s max=415.69ms p(90)=92.29ms p(95)=108.2ms<br />http_req_duration…………..: avg=119.37ms min=69.74ms med=107.3ms max=511.6ms p(90)=157.32ms p(95)=179.86ms<br />{ expected_response:true }…: avg=119.37ms min=69.74ms med=107.3ms max=511.6ms p(90)=157.32ms p(95)=179.86ms<br />http_req_failed…………….: 0.00% ✓ 0 ✗ 80<br />http_req_receiving………….: avg=145.3µs min=0s med=0s max=1.1ms p(90)=605.84µs p(95)=895.41µs<br />http_req_sending……………: avg=24.8µs min=0s med=0s max=481.1µs p(90)=95.75µs p(95)=136.96µs<br />http_req_tls_handshaking…….: avg=0s min=0s med=0s max=0s p(90)=0s p(95)=0s<br />http_req_waiting……………: avg=119.2ms min=69.74ms med=107.07ms max=511.6ms p(90)=157.28ms p(95)=179.86ms<br />http_reqs………………….: 80 4.965341/s<br />iteration_duration………….: avg=2.27s min=2.16s med=2.24s max=3.01s p(90)=2.33s p(95)=2.35s<br />iterations…………………: 40 2.48267/s<br />vus……………………….: 1 min=1 max=10<br />vus_max……………………: 10 min=10 max=10</pre><p>The average response time has increased slightly to 119.37 milliseconds, and no HTTP requests have failed. However, the maximum response time of the request is 511.6 milliseconds, whereas previously, it was only 108.36 milliseconds.</p><p>We need to check the application log to debug why the maximum response increases to 511.6 milliseconds. However, we have a lot of logs showing in the application pod. Let’s use Telepresence to debug this performance problem to ease the debugging task.</p><h4><strong>Install Telepresence to your Kubernetes Cluster</strong></h4><p>Telepresence can be installed on all operating systems. Run the following command to install Telepresence if you are using a macOS or Linux machine, then use the installation details below. If not, go to <a href="https://www.getambassador.io/docs/telepresence/latest/install?os=windows">Telepresence’s official installation page</a>.</p><pre>sudo curl -fL https://app.getambassador.io/download/tel2/linux/amd64/latest/telepresence -o /usr/local/bin/telepresence<br />sudo chmod a+x /usr/local/bin/telepresence</pre><p>To check whether Telepresence is installed successfully, run the telepresence version command. This will return an output similar to the one below:</p><pre>Client : v2.13.2</pre><p>To allow Telepresence to interact with the Kubernetes cluster, you need to install the Telepresence Traffic Manager to the Kubernetes cluster by running the telepresence helm install command. Then, run this command kubectl get pod — namespace ambassador to confirm that it has been installed correctly.</p><p>To connect your local machine to the Kubernetes cluster, run the telepresence connectcommand. You should see a similar output in the console as the one below:</p><pre>Launching Telepresence Daemon<br />Connected to context default (https://&lt;your-Kubernetes-cluster-public-IP&gt;)</pre><p>With Telepresence now installed and running, we can now use it to debug the blog app.</p><h4><strong>Using Telepresence to Debug the App</strong></h4><p>Before debugging the blog app, you need to intercept the requests sent to the blog app in the Kubernetes cluster to the blog app running on your local machine. To do this, run the following command:</p><pre>telepresence intercept blogapp - port 8081:8081 - env-file .env</pre><p>On doing this, telepresence will start the intercepting process for the blog service and store all the environment variables of the blog service to .env file. You should see the following output as below:</p><pre>Intercept name : blogapp<br />State : ACTIVE<br />Workload kind : Deployment<br />Destination : 127.0.0.1:8081<br />Service Port Identifier: http<br />Volume Mount Point : /tmp/telfs-230806768<br />Intercepting : all TCP requests</pre><p>Here, you see the traffic agent has been installed to intercept the blog service. Let’s debug the blog service from Visual Studio Code.</p><h4><strong>Debug the blog app from Visual Studio Code</strong></h4><p>First, you need to install delve to debug the Go code. You can do this by running the command below:</p><pre>go install github.com/go-delve/delve/cmd/dlv@latest</pre><p>Then, install Go’s extension in your Visual Studio Code</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/467/0*OZzID7pDa9qc_zY6" /><figcaption>Figure 1: Install the Go extension in Visual Studio Code</figcaption></figure><p>Inside the .vscode/launch.json file, you already have the configuration for debugging the blog service app.</p><pre>{<br />    &quot;version&quot;: &quot;0.2.0&quot;,<br />    &quot;configurations&quot;: [<br /><br />        {<br />            &quot;name&quot;: &quot;Launch with env file&quot;,<br /> &quot;type&quot;: &quot;go&quot;,<br /> &quot;request&quot;: &quot;launch&quot;,<br /> &quot;mode&quot;: &quot;debug&quot;,<br /> &quot;program&quot;: &quot;${workspaceRoot}&quot;,<br />        &quot;envFile&quot;: &quot;${workspaceFolder}/.env&quot;<br />        }<br />    ]<br />}</pre><p>Visual Studio Code will use the values in the .envfile to connect with the Kubernetes cluster and will run the blog app using the code files from the root directory of the application code.</p><p>To debug the blog app, set breaking points to the lines of code that you want the debugging process to stop. Let’s set the breaking points to line 90 and line 180 inside the handlers.go file since we want to check the code processes related to creating a new blog and removing a blog post.</p><p>Click on the “Run and Debug” button on the left panel of the screen and click on the “Play” button to run the “Launch with env file” debug task.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/515/0*WQ2IPe1Djc5B8g-q" /><figcaption>Figure 2: Run the debug by clicking on “Launch with env file”</figcaption></figure><p>Now that the debugging setup is done, let’s try to create a new blog post by running the following curl command:</p><pre>curl - location 'http://45.77.43.83:8081/blog' \<br /> - header 'Content-Type: application/json' \<br /> - data '{<br />&quot;body&quot;:&quot;this is a great blog one&quot;<br />}'</pre><p>You should see the debugger in Visual Studio Code stop at line 90 in the handlers.go file.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/0*9kp98qc0pgCz69QL" /><figcaption>Figure 3: Debugger jumps to line 90 in the handlers.go file</figcaption></figure><p>Looking carefully at the code inside the CreateBlog function, we don’t see anything wrong with it. So, let’s click on the “Continue” button in the Visual Studio Code app; we see that the curl process has stopped with the response containing information about the blog id.</p><pre>{<br />&quot;id&quot;: 786<br />}</pre><p>Let’s try to remove this blog id using the delete blog API by running the command below:</p><pre>curl - location - request DELETE 'http://45.77.43.83:8081/blog/786'</pre><p>The Visual Studio Code debugger now stops at line 180 in handlers.go file.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/843/0*4gATJalqs7I8wFHT" /><figcaption>Figure 4: Debugger jumps to line 180 in the handlers.go file</figcaption></figure><p>Carefully looking at the code inside the DeleteBlog function, you’ll see that the delete API will try to select all the existing blogs inside the blog table: rows, err:= config.DB.Query(&quot;SELECT * FROM blog&quot;) and then iterate the rows’ values using for rows.Next(). When iterating over the list of rows, if the blog id in the list does not equal the provided id in the URL of the delete API, the API will delete that blog.</p><p>Wait, we have a problem here. The API should delete the blog if the blog id in the list of blogs matches the provided ID in the API URL. This is why we have an API request that took more than 500 milliseconds to complete because the delete function will try to delete every blog except the one with the id provided in the API URL. Suppose we have 500 blogs in the database. The API will try to delete 499 blogs instead of just 1 blog.</p><p>Let’s click on the “Continue” button in the Visual Studio Code to complete this API request. Run the following command curl — location “http://45.77.43.83:8081” to check all the blogs we have now. To do this, replace the existing DeleteBlog function code with the code below.

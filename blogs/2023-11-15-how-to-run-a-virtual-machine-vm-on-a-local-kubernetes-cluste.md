<h4>Guest Author</h4><p>Kubernetes has emerged as the de facto standard for container orchestration because it revolutionized the way applications are deployed and managed at scale. But sometimes, it is a good choice to run a GPU-intensive workload, tokenization, or LDAP/Active Directory applications on virtual machines (VMs) instead of containers for security, compatibility, or performance reasons.</p><p>Virtual machines have long been the go-to technology for running diverse workloads, offering greater isolation, compatibility with legacy applications, and the ability to run operating systems of different types. However, managing VMs in a traditional infrastructure along with <a href="https://www.getambassador.io/docs/envoy/latest/concepts/cloudnative">containerized applications</a> on Kubernetes can be challenging and time-consuming. This is where Kubevirt steps in, offering a solution that allows you to run versatile and isolated virtual machines (VMs) inside your local <a href="https://www.getambassador.io/docs/edge-stack/latest/topics/concepts/kubernetes-network-architecture">Kubernetes cluster</a>.</p><p>In this article, we will explore how you can harness the power of Kubevirt to efficiently run virtual machines inside your local Kubernetes cluster, enabling you to embrace a hybrid cloud architecture and maximize the utilization of your infrastructure resources. We will dive into the concepts, tools, and best practices required to set up and manage VMs within a Kubernetes environment, providing you with the knowledge and guidance to embark on your VM orchestration journey. Whether you are a developer, an infrastructure engineer, or an IT professional looking to optimize your resource allocation and simplify your deployment pipelines, this article will serve as your comprehensive guide to running virtual machines on a local Kubernetes cluster. So, let’s get started.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*Mv46h3o5oOB2efTErKR8uA.png" /></figure><h4>What is Kubevirt?</h4><p>Kubevirt is an open-source solution created by a Redhat engineer that makes provisioning, managing, and controlling VMs inside Kubernetes alongside containerized applications effortless. It leverages the power of the Virtlet project to enable running VMs on Local Kubernetes clusters.<br />But why do we need Kubevirt?</p><p>Developers often require dedicated environments to test their applications, especially when dealing with complex setups that involve multiple services and dependencies. By utilizing KubeVirt in a <a href="https://www.getambassador.io/use-case/productive-local-dev-environment">local Kubernetes</a> cluster, developers can create VMs to simulate realistic testing environments. Apart from this, Kubevirt can also be used to deploy &amp; test legacy applications that run on Virtual Machine (VM) on your local cluster.</p><h4>Prerequisites (Setting up local Kubernetes cluster)</h4><p>Before getting started with kubeVirt and running VMs on our local Kubernetes cluster, we need to ensure we have a running local Kubernetes cluster already in place. In this article, we have used Minikube as our chosen option for running a local Kubernetes cluster; however, alternatives like K0s, K3s, or Kind are also popular choices. Remember installing minikube is essential for running the Kubernetes cluster locally. You can follow <a href="https://minikube.sigs.k8s.io/docs/start/">these instructions</a> to download &amp; install minikube on your machine. After installing minikube, you’d need to start it by running the command below on your command line interface(CLI):</p><pre>minikube start</pre><p>Minikube will, by default, create a cluster named “minikube” after you execute this command.</p><h4>Step 1: Installing Kubevirt — Enabling Kubevirt Addon</h4><p>You can enable Kubevirt addon in Minikube by using the following command</p><pre>minikube addons enable kubevirt</pre><p>Once the kubevirt addon is enabled, you can check if it’s working properly by running the following command</p><pre>minikube kubectl - get all -n kubevirt</pre><p>Note: There should be 7 pods, 3 services, 1 daemonset, 3 deployment apps, and 3 replica sets deployed. It might take a few minutes to deploy the following resources, depending on your network connection.</p><p>Once the resources are deployed &amp; running, you can move to step 2.</p><h4>Step 2: Installing Virtctl</h4><p>Virtctl is an additional library provided by kubevirt to handle start/stop applications on VMs and also to access serial and graphical ports of VMs. To install Virtctl run the following command:</p><pre>VERSION=$(kubectl get kubevirt.kubevirt.io/kubevirt -n kubevirt -o=jsonpath=&quot;{.status.observedKubeVirtVersion}&quot;)<br />ARCH=$(uname -s | tr A-Z a-z)-$(uname -m | sed 's/x86_64/amd64/') || windows-amd64.exe<br />echo ${ARCH}<br />curl -L -o virtctl https://github.com/kubevirt/kubevirt/releases/download/${VERSION}/virtctl-${VERSION}-${ARCH}<br />chmod +x virtctl<br />sudo install virtctl /usr/local/bin</pre><p>Now that we have installed Virtctl we are all set to create our first VM resource inside our local Kubernetes cluster.</p><h4>Step 3: Creating VM resource inside the local Kubernetes cluster</h4><p>Create a file named <em>VirtualMachine.yaml</em> and paste the YAML configuration below to it:</p><pre>apiVersion: kubevirt.io/v1<br />kind: VirtualMachine<br />metadata:<br />  name: testvm<br />spec:<br />  running: false<br />  template:<br />    metadata:<br />      labels:<br />        kubevirt.io/size: small<br />        kubevirt.io/domain: testvm<br />    spec:<br />      domain:<br />        devices:<br />          disks:<br />            - name: containerdisk<br />              disk:<br />                bus: virtio<br />            - name: cloudinitdisk<br />              disk:<br />                bus: virtio<br />          interfaces:<br />          - name: default<br />            masquerade: {}<br />        resources:<br />          requests:<br />            memory: 64M<br />      networks:<br />      - name: default<br />        pod: {}<br />      volumes:<br />        - name: containerdisk<br />          containerDisk:<br />            image: quay.io/kubevirt/cirros-container-disk-demo<br />        - name: cloudinitdisk<br />          cloudInitNoCloud:<br />            userDataBase64: SGkuXG4=</pre><p>You can also download the file using the following command:</p><pre>wget -o VirtualMachine.yaml https://raw.githubusercontent.com/kubevirt/kubevirt.github.io/master/labs/manifests/vm.yam</pre><p>Now that your Virtualmachine.yaml is ready, go to your terminal and apply the YAML configuration file into the minikube Kubernetes cluster using the command below:</p><pre>minikube kubectl - apply -f VirtualMachine.yaml</pre><p>Then run the command below to check the status of your VM.</p><pre>kubectl get vms</pre><h4>Step 4: Starting a Virtual Machine</h4><p>To start the virtual machine using Virtctl, run this command</p><pre>./virtctl start testvm</pre><p>Then proceed to confirm whether our VM is running or not using this command:</p><pre>minikube kubectl - get vms</pre><p>There are 3 statuses for the VMs, namely scheduling, scheduled, and running. Here’s a detailed explanation for each of them:</p><ul><li><strong>Scheduling:</strong> During this stage, the Kubevirt scheduler determines the appropriate node on which VMs can be scheduled based on the resource availability. It takes into account factors such as CPU, memory, storage, and network requirements specified for the VM.</li><li><strong>Scheduled:</strong> Once the VM is scheduled, the Kubernetes controller responsible for Kubevirt creates the necessary resources and configurations on the target node. This includes allocating storage volumes, setting up the network interfaces, and initializing the VM’s environment.</li><li><strong>Running:</strong> At this stage, the VM has started running on the assigned node. The Kubevirt controller monitors the VM’s status and ensures it’s running properly within the Kubernetes cluster.</li></ul><p><strong>Stopping the VMs: </strong>To stop VMs scheduled, you can use the following command</p><pre>./virtctl stop {VM_Name}</pre><p>Note: Replace {VM Name} with the VM name you wish to stop.</p><p><strong>Deleting the VMs: </strong>To delete the VMs you stopped, you can use the following command:</p><pre>kubectl delete vmis {VM_Name}</pre><p>Note: Replace {VM Name} with the VM name you wish to delete.</p><p><strong>Stopping Minikube: </strong>Now that you are done with the tutorial, you can stop the minikube instance by running the minikube stop command.</p><h3>Securing Your VMs</h3><p>Running VMs inside your <a href="https://www.getambassador.io/docs/cloud/latest/service-catalog/howtos/qs-java-advanced">local Kubernetes</a> cluster using Kubevirt introduces specific security considerations. Here are some essential security considerations you should follow.</p><h4><strong>Isolation and resource allocation:</strong></h4><ul><li><strong>Namespace isolation</strong>: Utilize Kubernetes namespaces to isolate VM workloads and prevent unauthorized access or data leakage between different namespaces.</li><li><strong>Resource quotas</strong>: Define resource quotas for VMs to prevent resource contention. Set limits on CPU, memory, and storage usage to avoid VMs monopolizing resources.</li></ul><h4><strong>Hypervisor security:</strong></h4><ul><li><strong>Secure hypervisor configuration</strong>: Ensure that the hypervisor used by Kubvirt is properly configured with security best practices. Follow guidelines provided by the specific hypervisor technology, such as KVM or QEMU.</li></ul><h3>Conclusion</h3><p>In this blog, we explored how you can run VMs (Virtual Machines) inside your local Kubernetes cluster using Kubevirt. By combining the power of Kubernetes’ container orchestration with the flexibility of VMs, you can unlock a range of possibilities for your applications.</p><p>As the world of technology evolves, the combination of VMs and Kubernetes will continue to play a crucial role in modern software development and deployment.
